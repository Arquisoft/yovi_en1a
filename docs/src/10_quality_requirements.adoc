=== Quality Scenarios

Each scenario follows the arc42 structure: source of stimulus, stimulus, environment, artifact, response, and response measure.

==== QS-01: Board updates are reflected in real time for all players

[cols="1,3", options="noheader"]
|===
| Source        | Player placing a tile during a multiplayer game session
| Stimulus      | A player places a tile on the board
| Environment   | System under normal load, two or more players in the same game room
| Artifact      | Game server and frontend rendering layer
| Response      | The placed tile appears on all other players' boards
| Response Measure | The board update is visible to all participants within 500 milliseconds in at least 95% of placements
|===

==== QS-02: System handles peak concurrent load

[cols="1,3", options="noheader"]
|===
| Source        | Multiple players starting game sessions simultaneously
| Stimulus      | 100 concurrent users are playing active games at the same time
| Environment   | Peak usage period
| Artifact      | Backend services and database layer
| Response      | All game sessions continue without interruption; no board state is lost or corrupted
| Response Measure | Error rate stays below 1%; average response time stays below 2 seconds
|===

==== QS-03: New user can start a game without prior instructions

[cols="1,3", options="noheader"]
|===
| Source        | First-time user with no prior knowledge of the system
| Stimulus      | User opens the application and attempts to start a game
| Environment   | Standard browser, no tutorial shown beforehand
| Artifact      | Frontend UI and onboarding flow
| Response      | The user finds the game lobby, joins or starts a game, and successfully places their first tile
| Response Measure | At least 80% of test users place their first tile within 3 minutes without external help
|===

==== QS-04: Player receives clear feedback when a tile placement is invalid

[cols="1,3", options="noheader"]
|===
| Source        | Player during an active game session
| Stimulus      | Player attempts to place a tile in a position not allowed by the rules
| Environment   | Normal gameplay, optional requirements
| Artifact      | Frontend validation and feedback layer
| Response      | The invalid placement is visually indicated
| Response Measure | No raw error message or silent failure occurs; feedback is shown within 200 milliseconds of the attempt
|===

==== QS-05: Turn is skipped automatically when the time limit expires
This is part of an optional improvement we will next implement:

[cols="1,3", options="noheader"]
|===
| Source        | Game timer running on the server
| Stimulus      | A player does not place a tile before their turn time limit expires
| Environment   | Active multiplayer or single-player game session
| Artifact      | Turn management logic in the game server
| Response      | The player's turn is skipped and the next player's turn begins without manual intervention
| Response Measure | The turn transition happens within 1 second of the timer expiring in 100% of cases; the board state remains consistent for all players
|===

==== QS-06: Application is available during expected usage hours

[cols="1,3", options="noheader"]
|===
| Source        | End users accessing the system at any time
| Stimulus      | User attempts to access the application
| Environment   | Non-maintenance period, standard operating conditions
| Artifact      | Deployment infrastructure and services
| Response      | The application loads and is fully functional
| Response Measure | System availability is at least 99% measured monthly, excluding scheduled maintenance windows
|===

==== QS-07: Unauthorized users cannot access protected resources

[cols="1,3", options="noheader"]
|===
| Source        | Unauthenticated or malicious external actor
| Stimulus      | Attempt to access player data or game state via the API without a valid session token
| Environment   | Any environment, including production
| Artifact      | API authentication and authorization layer
| Response      | The request is rejected with an appropriate HTTP error (401 or 403)
| Response Measure | Zero successful unauthorized accesses to protected endpoints in security audits
|===

==== QS-08: Passwords and sensitive data are stored securely

[cols="1,3", options="noheader"]
|===
| Source        | Developer or database administrator
| Stimulus      | Inspection of the data stored in the database
| Environment   | Any environment where user accounts exist
| Artifact      | User management service and persistence layer
| Response      | No plaintext passwords or sensitive tokens are found in storage.
| Response Measure | All passwords are hashed with a modern algorithm using bcrypt, no plaintext credentials exist in any table or log
|===

==== QS-09: A developer can add a new tile type with limited effort

For optional requirements and possible extensions, the application must be scalable.

[cols="1,3", options="noheader"]
|===
| Source        | Developer extending the system
| Stimulus      | A new tile shape or type needs to be added to the game
| Environment   | Development environment, working codebase
| Artifact      | Game logic module and tile configuration layer
| Response      | The new tile type appears correctly on the board and follows placement rules
| Response Measure | The change requires modifications in no more than 3 files and can be completed in under 2 hours by a developer familiar with the codebase
|===

==== QS-10: The test suite covers critical game logic

[cols="1,3", options="noheader"]
|===
| Source        | Development team running CI pipeline
| Stimulus      | A pull request is submitted with changes to the board or tile placement logic
| Environment   | Automated CI environment with GitHub Actions.
| Artifact      | Unit and integration test suite
| Response      | The pipeline runs all relevant tests and reports pass or fail clearly
| Response Measure | Code coverage for core game logic modules is at least 80%; the CI pipeline completes in under 10 minutes
|===
