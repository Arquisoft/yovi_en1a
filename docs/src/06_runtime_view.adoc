== Runtime View

This section describes the dynamic behavior of GAME Y. It illustrates how the decomposed building blocks cooperate to fulfill key user scenarios based on the implemented REST endpoints.

=== Scenario 1: User Authentication (Login & Registration)

*Description:* The entry point where a user creates an account or authenticates via the `RegisterForm`.
*Trigger:* User submits credentials. The component determines the endpoint based on the `mode` state ('login' or 'register').
*Endpoint(s):* `POST :3000/createuser` or `POST :3000/login`.

[plantuml, human-vs-ai-move, svg]
----
@startuml
autonumber
actor "Player" as User
participant "Webapp\n(RegisterForm.tsx)" as UI
participant "User Service\n(Port 3000)" as UserSvc
database "MongoDB" as DB

User -> UI : Enters Credentials
UI -> UserSvc : POST /login OR /createuser
activate UserSvc

UserSvc -> DB : findOne() OR insertOne()
activate DB
DB --> UserSvc : User Document / Success
deactivate DB

UserSvc -> UserSvc : bcrypt.compare() (if login)\nSign JWT (userId, username)

UserSvc --> UI : 200 OK { token, username, userId }
deactivate UserSvc

UI -> UI : localStorage.setItem('token', data.token)
UI -> User : Redirect to Lobby (?view=lobby)
@enduml
----

=== Scenario 2: Human Player vs. Bot (Game Loop)

*Description:* Orchestrating a move in "Human vs Bot" (hvb) mode.
*Trigger:* Player clicks a cell on the `GameBoard`.
*Endpoint(s):* `POST :3001/play/:gameId/move` and internal proxy to `POST :4000/v1/ybot/choose/:bot_id`.

[plantuml, target="game-loop-runtime", format="png"]
----
@startuml
autonumber
actor "Human Player" as User
participant "Webapp\n(GameBoard.tsx)" as UI
participant "Game API\n(Port 3001)" as API
participant "Logic Engine\n(Port 4000)" as Rust

User -> UI : Clicks Cell (x, y)
UI -> API : POST /play/:gameId/move\n{ player, x, y }
activate API

API -> API : updateWinStatus(s)
API -> Rust : POST /v1/ybot/choose/gamer_bot\n(YEN Layout JSON)
activate Rust

Rust -> Rust : Calculate AI Move
Rust --> API : 200 OK { coords: {x, y, z} }
deactivate Rust

API -> API : s.moves.push(botCoords)\nupdateWinStatus(s)
API --> UI : 200 OK (Updated SessionView)
deactivate API

UI -> User : Render updated grid
@enduml
----

=== Scenario 3: Exception Handling (Invalid Move)

*Description:* Handling illegal moves (e.g., clicking an occupied cell or playing out of turn).
*Trigger:* User attempts an invalid action.
*Endpoint(s):* `POST :3001/play/:gameId/move`.

[plantuml, target="error-handling-runtime", format="png"]
----
@startuml
autonumber
actor "Player" as User
participant "Webapp" as UI
participant "Game API" as API

User -> UI : Clicks occupied cell
UI -> API : POST /play/:gameId/move
activate API

API -> API : Check s.moves.some(...)
API --> UI : 400 Bad Request\n{ error: "Invalid move" }
deactivate API

UI -> UI : errorState = t('err_server')
UI -> User : Show Error Feedback
@enduml
----

=== Scenario 4: Match Conclusion and Statistics Persistence

*Description:* Saving game results to the permanent match history upon completion.
*Trigger:* `updateWinStatus(s)` sets state to 'finished'.
*Endpoint(s):* Internal call to `saveGameResult(session)`.

[plantuml, target="persistence-runtime", format="png"]
----
@startuml
autonumber
actor "Player" as User
participant "Game API" as API
database "MongoDB" as DB

User -> API : Final Winning Move
activate API

API -> API : updateWinStatus(s) == 'finished'
API -> DB : db.collection('games').insertOne({...})
activate DB
DB --> API : Acknowledge
deactivate DB

API --> User : 200 OK { status: 'finished', winner }
deactivate API

Note right of API: Session remains in Map\nuntil manually deleted or rematch.
@enduml
----

=== Scenario 5: Profile & Avatar Management

*Description:* Fetching match history and updating user personalization.
*Trigger:* User navigates to `?view=profile` or selects a new avatar.
*Endpoint(s):* `GET :3001/profile` and `POST :3001/profile/avatar`.

[plantuml, target="profile-runtime", format="png"]
----
@startuml
autonumber
actor "Player" as User
participant "Webapp\n(Profile.tsx)" as UI
participant "Game API\n(Port 3001)" as API
database "MongoDB" as DB

User -> UI : Clicks "Profile"
UI -> API : GET /profile (Auth: Bearer token)
activate API

API -> DB : games.find({ userId }).sort({ finishedAt: -1 })
API -> DB : users.findOne({ _id: userId })
DB --> API : Data Sets
API --> UI : 200 OK { winRate, matchHistory, avatarUrl }
deactivate API

User -> UI : Selects New Avatar
UI -> API : POST /profile/avatar\n{ avatarUrl: "filename" }
activate API
API -> DB : users.updateOne({ _id: userId }, { $set: { avatarUrl } })
API --> UI : 200 OK { success: true }
deactivate API
@enduml
----
