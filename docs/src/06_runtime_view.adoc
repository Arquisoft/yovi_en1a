== Runtime View

=== Scenario 1: User Authentication and Login Flow (Entry)

*Description:* Secure login process using **bcrypt** for credential verification and **JWT** for session issuance. This is the entry point for all authenticated interactions.

*Notable Aspects:* This interaction occurs strictly between the Webapp and the User Service (Port 3000). Password safety is ensured via hashing, and the resulting stateless token allows for decentralized authorization in subsequent game calls.

[plantuml, target="auth-flow", format="png"]
----
@startuml
autonumber
actor "Human Player" as User
participant "Webapp\n(React)" as UI
participant "User Service\n(Node.js:3000)" as UserSvc
database "MongoDB" as DB

User -> UI : Submits credentials\n(username, password)
UI -> UserSvc : POST /auth/login
activate UserSvc

UserSvc -> DB : Find user by username
activate DB
DB --> UserSvc : User Document (Hashed PWD)
deactivate DB

UserSvc -> UserSvc : bcrypt.compare(password, hash)
UserSvc -> UserSvc : Sign JWT (userId, username)

UserSvc --> UI : 200 OK (JWT Token)
deactivate UserSvc

UI -> UI : Store JWT in SessionStorage
UI -> User : Redirect to Game Lobby
@enduml
----

=== Scenario 2: Human Player vs. AI (Game Loop)

*Description:* The primary game loop where `gameyapi` orchestrates moves between the user and the Rust Engine.

*Notable Aspects:* Every move request is authorized via JWT. `gameyapi` (Port 3001) maintains the active session in-memory and proxies the move to the Logic Engine (Port 4000) for validation and AI counter-move calculation.

[plantuml, target="human-vs-ai-move", format="png"]
----
@startuml
autonumber
actor "Human Player" as User
participant "Webapp\n(React)" as UI
participant "gameyapi\n(Node.js:3001)" as API
participant "Logic Engine\n(Rust:4000)" as Rust

User -> UI : Clicks hexagonal cell
UI -> API : POST /game/move\n(JWT + YEN Payload)
activate API

API -> API : Verify JWT Statelessly
API -> Rust : POST /next-move\n(YEN Payload)
activate Rust

Rust -> Rust : Validate Player Move
Rust -> Rust : Calculate AI Response
Rust --> API : 200 OK (New YEN Board State)
deactivate Rust

API -> API : Update in-memory session map
API --> UI : 200 OK (New Board State)
deactivate API

UI -> User : Render updated grid
@enduml
----

=== Scenario 3: Invalid Move Validation (Exception Flow)

*Description:* System resilience during rule violations. This scenario occurs when the user attempts a move that contradicts the game logic.

*Notable Aspects:* The **Logic Engine** acts as the single source of truth. If a move is invalid (e.g., cell occupied), the engine returns a 400-range error which `gameyapi` forwards to the UI to trigger a user notification.

[plantuml, target="invalid-move", format="png"]
----
@startuml
autonumber
actor "Player" as User
participant "Webapp\n(React)" as UI
participant "gameyapi\n(Node.js:3001)" as API
participant "Logic Engine\n(Rust:4000)" as Rust

User -> UI : Clicks occupied cell
UI -> API : POST /game/move (JWT + YEN)
activate API

API -> Rust : POST /next-move
activate Rust
Rust -> Rust : Rule Check: OCCUPIED_CELL
Rust --> API : 400 Bad Request (Error: "INVALID_MOVE")
deactivate Rust

API --> UI : 400 Bad Request (Error: "INVALID_MOVE")
deactivate API

UI -> UI : Revert optimistic UI state
UI -> User : Show Warning: "Cell already occupied!"
@enduml
----

=== Scenario 4: Match Conclusion and Statistics Persistence (Exit)

*Description:* The data flow triggered when a terminal state is reached (Win/Loss/Draw), moving data from volatile memory to the database.

*Notable Aspects:* Once the **Logic Engine** detects a game-over state, `gameyapi` triggers the persistence. Finished games are written to MongoDB to update the player's permanent history and statistics.

[plantuml, target="match-conclusion", format="png"]
----
@startuml
autonumber
actor "Player" as User
participant "Webapp\n(React)" as UI
participant "gameyapi\n(Node.js:3001)" as API
participant "Logic Engine\n(Rust:4000)" as Rust
database "MongoDB" as DB

User -> UI : Makes the winning move
UI -> API : POST /game/move (JWT + YEN)
activate API

API -> Rust : POST /next-move
activate Rust
Rust --> API : Result: "WINNER_PLAYER" (YEN JSON)
deactivate Rust

API -> DB : Insert finished match record
API -> DB : Update User Stats ($inc wins)
API -> API : Remove session from in-memory Map

API --> UI : 200 OK (Final Result + Updated Stats)
deactivate API

UI -> User : Display "Victory!" and new stats
@enduml
----
