== Risks and Technical Debts

The following table lists the identified technical risks and technical debts within the YOVI_0 architecture. A Risk describes a potential future problem (e.g., security vulnerabilities or system overload), while a Technical Debt represents incomplete implementations or shortcuts taken during development that need to be refactored before a production release. The items are ordered by their priority and potential impact on the system.

[cols="1,1,3,4", options="header"]
|===
| Priority | Type | Name & Description | Mitigation Strategy / Next Steps

| High
| Risk
| **Missing Rate Limiting on Public API** +
The system exposes a public REST API for external bots. Currently, there is no rate limiting implemented. This makes the system highly susceptible to abuse, spam, or accidental Denial of Service (DoS) attacks from malfunctioning third-party bots.
| Implement rate-limiting middleware (e.g., `express-rate-limit`) on the Node.js services or throttle requests via an Nginx reverse proxy before production deployment.

| High
| Risk
| **Permissive CORS Policy** +
The GameY API (`gameyapi`) currently uses a permissive CORS policy (`cors()` with no origin restriction) to easily facilitate external bot integration during development. This allows any domain to send requests to the logic engine.
| Tighten the CORS configuration by implementing a strict origin whitelist for the GameY API, similar to the `ALLOWED_ORIGINS` setup used in the User Service.

| Medium
| Debt
| **Runtime Secrets Management** +
While sensitive configuration data is securely stored in GitHub Repository Secrets for the CI/CD pipeline, they are ultimately injected as plain environment variables into the production VM via docker-compose.yml. While this effectively prevents source-code leaks, it leaves the runtime environment slightly vulnerable if the host machine is compromised.
| Migrate the runtime environment to use native Docker Secrets or a vault-based solution for the production hardening phase, ensuring secrets are never exposed in plain text on the host filesystem.

| Medium
| Debt
| **Lack of Real-time Push (WebSockets)** +
The "Human vs. Human" multiplayer scenario currently relies on REST API calls. Asynchronous push communication (WebSockets) is planned but not yet implemented. Relying on client-side polling for multiplayer turns is inefficient and degrades the User Experience.
| Integrate WebSockets (e.g., via Socket.IO) into the User Service to actively push game state updates directly to connected React clients.

| Low
| Debt
| **Decentralized Logging** +
All services currently emit logs exclusively to stdout/stderr. While Docker captures these, there is no centralized log aggregation. This makes debugging difficult in a multi-container environment.
| Deploy a centralized log aggregation stack (e.g., shipping Docker logs to Loki, integrated with the existing Grafana setup) in a future iteration.

|===
