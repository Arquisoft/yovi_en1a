== 4. Solution Strategy

The core architectural style (Microservices), the primary technology stack (Rust, React.js, Node.js), and the communication protocol (YEN Notation) were predefined as strict constraints for this project (as detailed in Sections 2 and 3). 

Therefore, our solution strategy focuses exclusively on the actual architectural and organizational decisions our team made to complete and integrate the system.

=== 4.1. Technology Decisions

|===
| Decision | Technology | Motivation
| **Database** | MongoDB | Document-based storage naturally fits the predefined YEN Notation (JSON) format seamlessly. It allows for flexible player statistics and game history storage without the need for rigid schema migrations.
|===

=== 4.2. Achievement of Quality Goals

* **Flexibility & Persistence:** By deliberately choosing **MongoDB** as our database, we ensure that as the game evolves (e.g., adding new bot metrics, different match data structures), the system can dynamically adapt to store these complex JSON objects efficiently.
